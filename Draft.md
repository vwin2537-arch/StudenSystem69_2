สร้างระบบวางแผนวันปฏิบัติงาน เช็คชื่อ และรายงานผลการปฏิบัติงาน อุทยานแห่งชาติเอราวัณ (Draft โปรเจค 2)
 1. ส่วนภาพรวมและเทคโนโลยี (Overview & Tech Stack) สร้าง Web Application แบบ Single Page Application (SPA) สำหรับ "ระบบวางแผนวันปฏิบัติงาน เช็คชื่อ และรายงานผลการปฏิบัติงาน อุทยานแห่งชาติเอราวัณ" โดยใช้ Google Apps Script เป็น Backend และ Google Sheets เป็นฐานข้อมูลหลัก ระบบต้องรองรับการทำงานแบบเจ้าหน้าที่วางแผนวันทำงานล่วงหน้า 30 วันภายในรอบงาน 90 วัน แล้วส่งให้แอดมินอนุมัติก่อนจึงจะสามารถเช็คชื่อเข้างานได้ตามวันที่อยู่ในแผนที่อนุมัติแล้ว
 Frontend: HTML5, CSS3, JavaScript
 Framework & Libraries: Tailwind CSS (สำหรับจัด Layout), SweetAlert2 (สำหรับแจ้งเตือน), Chart.js (สำหรับกราฟแดชบอร์ด), DataTables (สำหรับตารางข้อมูล)
 Typography: ใช้ฟอนต์จาก Google Fonts เช่น `Kanit` หรือ `Sarabun` เป็นฟอนต์หลักของระบบ เพื่อให้รองรับภาษาไทยได้สวยงาม อ่านง่าย และภาพรวมของระบบดูทันสมัยมากขึ้น
 Storage: รูปภาพแนบการปฏิบัติงานให้บันทึกลง Google Drive และเก็บลิงก์ลง Google Sheet
 Architecture: ใช้หน้าเว็บเดียวที่สลับมุมมองตามสิทธิ์ผู้ใช้ พร้อมมี state management ฝั่ง frontend สำหรับข้อมูลผู้ใช้ แผนวันทำงาน สถานะเช็คชื่อ และข้อมูลแอดมิน
 การจัดการโฟลเดอร์รูปภาพ: ให้ระบบจัดเก็บรูปถ่ายการปฏิบัติงานใน Google Drive แบบแยกโครงสร้างโฟลเดอร์อัตโนมัติเป็น "โฟลเดอร์เดือน > โฟลเดอร์วันที่ > โฟลเดอร์รายผู้ใช้" เพื่อให้ค้นหาไฟล์ย้อนหลังได้ง่าย เป็นระเบียบ และรองรับการตรวจสอบย้อนหลังของแอดมิน
 2. ส่วนโครงสร้างฐานข้อมูล (Database & Storage) กำหนดให้ใช้ Google Sheet ID: https://docs.google.com/spreadsheets/d/1yRrEV40jMMQCzycruoj8lIM0NPXTRhxXxUfvsaTO2uM/edit?usp=sharing และ Google Drive Folder ID: https://drive.google.com/drive/folders/1kch--KBTd15dXHVIKutMAFy7Hj5ARtBo?usp=sharing โดยให้ระบบสร้างชีต (Tab) อัตโนมัติหากยังไม่มี และตรวจสอบว่าคอลัมน์ครบเสมอ ดังนี้:
 ชีต "Users": ID, Username, Password, Name, Role, Status
 หมายเหตุ: คอลัมน์ `Status` ใช้จัดการสถานะผู้ใช้ โดยมี 3 ค่า: `Unregistered` (Admin สร้างให้แล้วแต่ยังไม่ตั้งรหัสผ่าน), `Pending` (ลงทะเบียนแล้วรออนุมัติ), `Active` (อนุมัติแล้ว เข้าใช้งานได้)
 ชีต "AttendanceLog": LogID, Date, Name, Time_In, Time_Out, Task_Report, Photo_URL, Status, Latitude, Longitude, Distance_m, Selfie_URL
 ชีต "WorkCycles": CycleID, UserID, Name, Start_Date, End_Date, Required_Work_Days, Status
 ชีต "WorkPlans": PlanID, Submission_ID, CycleID, UserID, Name, Plan_Date, Plan_Status, Notes, Completed_LogID, Created_At, Submitted_At, Approved_At
 ชีต "ScheduleRequests": ReqID, CycleID, UserID, Name, Original_Date, Requested_Date, Reason, Status, Created_At, Decision_At
 หมายเหตุด้านข้อมูล: รอบงานมาตรฐานมีความยาว 90 วัน และจำนวนวันปฏิบัติงานที่ต้องเลือกคือ 30 วันต่อรอบ โดยระบบต้องรองรับการเพิ่มคอลัมน์ใหม่ได้โดยไม่ทำให้ระบบพัง
 หมายเหตุด้านรูปภาพ: คอลัมน์ `Photo_URL` ในชีต AttendanceLog ให้เก็บเป็นลิงก์ของโฟลเดอร์รูปภาพประจำวันของเจ้าหน้าที่คนนั้น ไม่ใช่ลิงก์ไฟล์รูปเดี่ยว เพื่อให้สามารถเก็บหลายรูปในวันเดียวกันได้อย่างเป็นระบบ
 3. ส่วนระบบสิทธิ์ผู้ใช้งาน (User Roles & Authentication) ให้มีหน้าระบบ Login เพื่อแบ่งสิทธิ์การใช้งานเป็น 2 ระดับ พร้อมใช้ session token สำหรับรักษาสถานะการเข้าสู่ระบบ:
 User ทั่วไป (เจ้าหน้าที่): สามารถเข้าถึงหน้าเช็คชื่อเข้างาน, หน้ารายงานผลปฏิบัติงาน (เช็คเอาท์), และหน้าแผนวันปฏิบัติงานของตนเอง
 Admin (หัวหน้างาน): สามารถเข้าถึงหน้าแดชบอร์ด, จัดการรอบงาน, อนุมัติแผนวันปฏิบัติงาน, อนุมัติคำร้องขอสลับวัน, และตรวจสอบรายงานประจำวัน
 ระบบต้องมีบัญชี Admin เริ่มต้นอัตโนมัติหากในชีต Users ยังไม่มีผู้ใช้ที่มีสิทธิ์ Admin (Status = Active)
 ระบบลงทะเบียนผู้ใช้ (User Registration Flow):
 - Admin สร้างผู้ใช้ใหม่โดยกำหนด Username และชื่อ → สถานะเริ่มต้น = `Unregistered`
 - ผู้ใช้ไปที่หน้า Login กดลิงก์ "ลงทะเบียน" → เลือกชื่อตนเองจาก Dropdown → ตั้งรหัสผ่าน + ยืนยันรหัสผ่าน → สถานะเปลี่ยนเป็น `Pending`
 - Admin กดอนุมัติ → สถานะเปลี่ยนเป็น `Active` → ผู้ใช้สามารถ Login เข้าระบบได้
 - Admin สามารถกดปฏิเสธ → รหัสผ่านถูกลบ สถานะกลับเป็น `Unregistered` ผู้ใช้ต้องลงทะเบียนใหม่
 - เฉพาะผู้ใช้ที่มีสถานะ `Active` เท่านั้นจึงจะ Login ได้ หากยังเป็น `Pending` ระบบจะแจ้ง "รอการอนุมัติ"
 - รหัสผ่านต้องมีความยาวอย่างน้อย 4 ตัวอักษร
 4. ส่วนฟังก์ชันการทำงานหน้าบ้าน (Frontend Features) สำหรับ User ทั่วไป (เมนูมี 3 แท็บ):
 โครงสร้าง Navigation: ต้องมี Top Navigation Bar ที่ดูทันสมัย แสดงชื่อระบบ ชื่อผู้ใช้งาน และปุ่มออกจากระบบอย่างชัดเจน โดยบนหน้าจอมือถือให้ยุบเมนูเป็นปุ่ม Hamburger เพื่อเปิด/ปิดเมนูการใช้งาน
 หน้าเช็คชื่อเข้างาน: มีปุ่มกด "ลงเวลาเข้างาน" โดยระบบจะอนุญาตให้กดได้เฉพาะวันที่อยู่ในแผนที่ได้รับการอนุมัติแล้วเท่านั้น และบันทึกวันเวลาปัจจุบันอัตโนมัติ
 หน้ารายงานผลปฏิบัติงาน (ลงเวลาออก): ฟอร์มให้กรอกรายละเอียด "หน้าที่ที่ทำประจำวัน" พร้อมช่องอัปโหลดไฟล์ "รูปถ่ายการปฏิบัติงาน" อย่างน้อย 3 รูป มี Preview รูปก่อนส่ง และเมื่อกดส่ง ระบบจะบันทึกเวลาออกงานพร้อมรายงานผล
 หน้าแผนวันปฏิบัติงาน: แสดงสรุปรอบงานปัจจุบัน จำนวนวันที่ต้องทำ จำนวนวันที่เลือกแล้ว จำนวนวันที่อนุมัติแล้ว และมีปฏิทินหรือรายการวันที่ให้ผู้ใช้เลือกวันทำงานล่วงหน้า 30 วันภายในรอบงาน จากนั้นกดส่งแผนเพื่อรออนุมัติ หากมีแผนรออนุมัติอยู่แล้วต้องไม่ให้ส่งแผนซ้ำ
 ฟังก์ชันเพิ่มเติมสำหรับ User: หากมีเหตุจำเป็นต้องหยุดหรือเปลี่ยนวัน ให้สามารถยื่นคำร้องขอสลับวันปฏิบัติงาน โดยเลือกวันเดิม วันที่ต้องการเปลี่ยน และระบุเหตุผล พร้อมแสดงประวัติคำร้องของตนเอง
 สำหรับ Admin (เมนูจัดการ):
 จัดการผู้ใช้ (User Management): แสดงสรุปจำนวนผู้ใช้แต่ละสถานะ (Active / Unregistered / Pending / ทั้งหมด), ฟอร์มเพิ่มผู้ใช้ใหม่ (Username + ชื่อ-นามสกุล), ตารางรายชื่อผู้ใช้ที่รออนุมัติพร้อมปุ่มอนุมัติ/ปฏิเสธ, และตารางผู้ใช้ทั้งหมดพร้อม Badge สถานะสี
 Dashboard: แสดงการ์ดสถิติภาพรวม เช่น จำนวนผู้มาปฏิบัติงานวันนี้ จำนวนรอบงานที่กำลังดำเนินการ จำนวนคำร้องที่รอดำเนินการ และกราฟ Chart.js สรุปสถิติการมาปฏิบัติงานรายเดือน
 จัดการรอบงาน: มีฟอร์มสำหรับสร้างรอบงานให้เจ้าหน้าที่แต่ละคน โดยกรอกวันเริ่มรอบเพียงครั้งเดียว และระบบคำนวณวันสิ้นสุดรอบ 90 วันให้อัตโนมัติ แต่สามารถรองรับการระบุวันสิ้นสุดเองได้ถ้าจำเป็น
 อนุมัติแผนวันทำงาน: ตารางแสดงชุดแผนวันทำงานที่ "รออนุมัติแผน" โดย grouped ตาม Submission_ID พร้อมปุ่มกด อนุมัติ / ไม่อนุมัติ และแสดงจำนวนวันที่ยื่นในชุดนั้น
 อนุมัติคำร้องขอสลับวัน: ตารางแสดงคำร้องที่รอดำเนินการ พร้อมปุ่มกด อนุมัติ / ไม่อนุมัติ และเมื่ออนุมัติแล้วต้องปรับสถานะวันเดิมและวันใหม่ให้สอดคล้องกัน
 ตรวจสอบรายงานปฏิบัติงาน: ตารางแสดงผลการปฏิบัติงานรายวันของเจ้าหน้าที่ พร้อมปุ่มกดดูรูปถ่ายที่แนบมา และสามารถเปิดดูรูปใน Modal Popup ได้
 5. ส่วนฟังก์ชันหลังบ้าน (Backend - Code.gs) เขียนฟังก์ชันสำหรับจัดการข้อมูลดังนี้:
 doGet(e) สำหรับแสดงผลหน้าเว็บ
 initializeApp(), login(), logout(), getSessionUser() สำหรับการเริ่มระบบและจัดการ session ผู้ใช้
 ensureSetup_() และ ensureSheet_() สำหรับสร้างชีตและตรวจสอบโครงสร้างคอลัมน์อัตโนมัติ รวมถึงสร้าง Admin เริ่มต้นหากยังไม่มี
 getUserAppData() และ getAdminAppData() สำหรับดึงข้อมูลทั้งหมดที่จำเป็นต่อการแสดงผลหน้า User และ Admin
 createWorkCycle() สำหรับสร้างรอบงาน 90 วัน
 createWorkPlan() สำหรับบันทึกแผนวันทำงานหลายวันในครั้งเดียว และผูกเป็นชุดด้วย Submission_ID
 updateWorkPlanApprovalStatus() สำหรับอนุมัติหรือไม่อนุมัติแผนวันทำงานทั้งชุด
 createScheduleChangeRequest() และ updateScheduleRequestStatus() สำหรับสร้างและจัดการคำร้องขอสลับวันปฏิบัติงาน
 checkIn(token, latitude, longitude, selfieData) สำหรับบันทึกเวลาเข้างาน โดยต้องตรวจสอบพิกัด GPS ว่าอยู่ในรัศมีที่กำหนด ตรวจสอบว่าเป็นวันที่อยู่ในแผนที่อนุมัติแล้ว บังคับถ่ายเซลฟี่ยืนยันตัวตน อัปโหลดเซลฟี่ไป Google Drive และบันทึกพิกัด+ลิงก์เซลฟี่ลง AttendanceLog
 uploadSelfie_(userName, dateStr, selfieData) — อัปโหลดรูปเซลฟี่เช็คอินไป Google Drive และคืน thumbnail URL สำหรับแสดงผลใน Dashboard
 checkOut() สำหรับบันทึกเวลาออกงาน รายงานผล และรูปภาพแนบ พร้อมเชื่อมโยงกับบันทึกการเข้างานของวันนั้น
 ฟังก์ชันรับไฟล์รูปภาพแบบ Base64 ถอดรหัสและบันทึกลง Google Drive อัตโนมัติ พร้อมตั้งค่าการแชร์ไฟล์เป็นสาธารณะ ("Anyone with the link can view") และคืนค่า URL กลับมาบันทึกลง Sheet
 calculateDistance_(lat1, lng1, lat2, lng2) — คำนวณระยะทางระหว่าง 2 พิกัดด้วยสูตร Haversine (หน่วยเมตร)
 ฟังก์ชันช่วยเหลือเพิ่มเติมสำหรับการคำนวณวันในรอบงาน, การนับวันที่ปฏิบัติงานแล้ว, การ normalize สถานะแผน, การตรวจสอบสิทธิ์การเช็คชื่อ, และการสรุปข้อมูลเพื่อแสดงผลหน้าเว็บ
 ฟังก์ชันระบบลงทะเบียนผู้ใช้:
 - getUnregisteredUsers() — ดึงรายชื่อผู้ใช้ที่ยังไม่ลงทะเบียน (public, ไม่ต้อง login)
 - registerUser(userId, password) — ผู้ใช้เลือกชื่อ + ตั้งรหัสผ่าน → เปลี่ยนสถานะเป็น Pending
 - addUser(token, username, name) — Admin สร้าง User ใหม่ → สถานะ Unregistered
 - approveUser(token, userId) — Admin อนุมัติ → สถานะ Active
 - rejectUser(token, userId) — Admin ปฏิเสธ → ลบรหัสผ่าน กลับเป็น Unregistered
 - getUsers(token) — Admin ดึงรายชื่อผู้ใช้ทั้งหมด (ไม่แสดง Password)
 ฟังก์ชันเพิ่มประสิทธิภาพ (Performance Optimization):
 - loginAndGetData(username, password) — รวม login + ดึงข้อมูลทั้งหมดในการเรียก API ครั้งเดียว (ลด round-trip)
 - restoreSessionAndGetData(token) — รวม session check + ดึงข้อมูลในครั้งเดียว
 - createWorkPlan ใช้ batch setValues() แทน appendRow loop เพื่อเขียนข้อมูลหลายแถวในครั้งเดียว (~10x เร็วขึ้น)
 ฟังก์ชันจัดการโฟลเดอร์รูปภาพ: ต้องมี helper สำหรับสร้างหรือค้นหาโฟลเดอร์เดือน, โฟลเดอร์วันที่, และโฟลเดอร์รายผู้ใช้โดยอัตโนมัติ รวมถึงกำหนดชื่อโฟลเดอร์ให้เป็นมาตรฐาน เช่น `2026-03`, `2026-03-30`, และชื่อผู้ใช้หรือชื่อเจ้าหน้าที่ เพื่อให้โครงสร้างไฟล์ใน Google Drive เป็นรูปแบบเดียวกันทั้งระบบ
 6. ส่วนกติกาธุรกิจ (Business Rules) ระบบต้องยึดกติกาสำคัญดังนี้:
 รอบงานมาตรฐาน 1 รอบ = 90 วัน
 เจ้าหน้าที่ต้องเลือกวันปฏิบัติงาน 30 วันต่อรอบ
 จะเช็คชื่อเข้างานได้เฉพาะวันที่อยู่ในแผนที่ได้รับการอนุมัติแล้ว และต้องอยู่ในรัศมีไม่เกิน 1 กิโลเมตรจากจุดลงเวลา (พิกัด 14.37462, 99.14541)
 ระบบตรวจสอบพิกัด GPS ผ่าน HTML5 Geolocation API (enableHighAccuracy) และคำนวณระยะทางด้วยสูตร Haversine บันทึกพิกัดและระยะทางลง AttendanceLog เพื่อตรวจสอบย้อนหลัง
 หากผู้ใช้ปิด GPS หรือปฏิเสธการเข้าถึงตำแหน่ง ระบบจะแจ้งข้อความที่ชัดเจนให้เปิด GPS แล้วลองใหม่
 หากยังไม่ส่งแผน หรือแผนยังไม่อนุมัติ ต้องไม่สามารถเช็คชื่อได้
 หากมีแผนที่รออนุมัติอยู่แล้ว ต้องไม่สามารถส่งแผนใหม่ซ้ำได้
 การขอสลับวันต้องอ้างอิงจากวันเดิมที่อยู่ในแผน และวันใหม่ต้องอยู่ภายในรอบงานเดียวกัน
 เมื่อแอดมินอนุมัติคำร้องขอสลับวัน ระบบต้องปรับสถานะวันเดิมและวันใหม่ให้สอดคล้องกัน
 ระบบต้องคงการทำงานเดิมของการอัปโหลดรูปลง Google Drive และการบันทึกลิงก์รูปไว้ครบถ้วน
 ระบบต้องจัดเก็บรูปภาพในโครงสร้างโฟลเดอร์แบบ "เดือน > วันที่ > ผู้ใช้" ภายใต้ Google Drive Folder หลักของโครงการ เพื่อให้การค้นหาและตรวจสอบย้อนหลังทำได้ง่าย
 ระบบต้องบังคับให้มีรูปแนบอย่างน้อย 3 รูปต่อการรายงานผล 1 ครั้ง และควรมีการปรับขนาดหรือบีบอัดรูปฝั่ง frontend ให้เหมาะสมก่อนอัปโหลด โดยใช้ maxWidth = 1024px และ quality = 0.65 ซึ่งเป็นขนาดที่สมดุลระหว่างความเร็วในการอัปโหลดและความชัดเพียงพอสำหรับการพิมพ์ลงรายงาน A4
 เมื่ออัปโหลดรูปสำเร็จ ระบบต้องบันทึกรูปทั้งหมดไว้ในโฟลเดอร์รายวันของผู้ใช้ และเก็บลิงก์โฟลเดอร์นั้นกลับไปที่ AttendanceLog เพื่อใช้เปิดดูรูปย้อนหลังจากหน้า Admin ได้สะดวก
 7. ส่วนการออกแบบ UI/UX (Design Guidelines)
 โทนสี: ธีมธรรมชาติ ป่าไม้ (โทนสีเขียวเอิร์ธโทน - น้ำตาลอ่อน) ให้เข้ากับบริบทอุทยานแห่งชาติ
 ความเข้ากันได้: ออกแบบเป็น Mobile-First เพื่อให้เจ้าหน้าที่ใช้งานผ่านสมาร์ทโฟนขณะปฏิบัติพื้นที่ได้สะดวก และรองรับบน PC สำหรับ Admin
 รูปแบบการนำทาง: ใช้ Navigation Bar แบบ sticky ด้านบน พร้อมปุ่ม Hamburger สำหรับหน้าจอขนาดเล็ก เมนูต้องเปิดใช้งานง่าย เห็นสถานะเมนูปัจจุบันชัดเจน และสลับแท็บได้ลื่นไหล
 ตัวอักษรและภาพรวมสมัยใหม่: ใช้ฟอนต์ `Kanit` หรือ `Sarabun` ทั่วทั้งระบบ ร่วมกับ card layout, rounded corners, soft shadow, spacing ที่โปร่งอ่านง่าย และปุ่มที่มีขนาดเหมาะกับการกดบนมือถือ
 การแจ้งเตือน: ใช้ SweetAlert2 ทั้งหมด เช่น "กำลังเข้าสู่ระบบ...", "กำลังบันทึกข้อมูล...", "ส่งแผนวันทำงานสำเร็จ", "อนุมัติแผนแล้ว", "ยื่นคำร้องสำเร็จ"
 ความเสถียร: ต้องมีการดัก error ทั้งระดับ global และระดับฟังก์ชัน render เพื่อป้องกันปัญหาหน้าขาวหลังล็อกอิน และควรมี fallback error message ที่ผู้ใช้มองเห็นได้ชัดเจน
 ประสบการณ์ใช้งานรูปภาพ: หน้า preview และอัปโหลดรูปควรใช้งานง่ายบนมือถือ รองรับการเลือกหลายรูปในครั้งเดียว แสดงจำนวนรูปที่เลือก และแจ้งเตือนทันทีหากรูปไม่ครบตามจำนวนขั้นต่ำ
 Logo: ใช้ตราสัญลักษณ์อุทยานแห่งชาติเอราวัณ (ช้าง 3 เศียร) เป็น Logo ระบบ โดย host รูปบน Google Drive (File ID: 1esgwz9mio7Y7YSLxrG2oXRpqsrY_2w4S) และใช้ URL สั้น `https://lh3.googleusercontent.com/d/{FILE_ID}=s{SIZE}` แสดงในหน้า Login (200px) และ Navbar (64px)
 Footer: ใส่ข้อความ "Copyright © 2026 ระบบรายงานการปฏิบัติงาน อุทยานแห่งชาติเอราวัณ"
 Cache Management: ใช้ APP_VERSION สำหรับ cache busting เมื่อมีการอัปเดตระบบ เพิ่ม meta tags (Cache-Control, Pragma, Expires) เพื่อป้องกันหน้าเก่าค้าง เมื่อ version เปลี่ยนระบบจะ clear token อัตโนมัติให้ผู้ใช้ login ใหม่
 Concurrent Users: ระบบออกแบบให้รองรับผู้ใช้งานพร้อมกัน ~15 คน โดยลด API calls ให้น้อยที่สุด (combined functions), ใช้ batch writes, และบีบอัดรูปก่อนอัปโหลด
 8. ส่วนผลลัพธ์ที่คาดหวัง (Expected Output)
 โค้ดสำหรับไฟล์ Code.gs (Backend ทั้งหมด)
 โค้ดสำหรับไฟล์ index.html (Frontend ทั้งหมด รวมถึง CSS และ JavaScript)
 โค้ดที่พร้อมใช้งานจริงทันทีใน Google Apps Script โดยไม่ขาดฟังก์ชันหลัก
 คำแนะนำสั้นๆ ในการตั้งค่า Google Sheet, Google Drive, การสร้างชีตเริ่มต้น, และวิธีการ Deploy Web App
 แนวทางทดสอบระบบหลัง deploy เช่น การล็อกอิน, การสร้างรอบงาน, การส่งแผนวันทำงาน, การอนุมัติแผน, การเช็คชื่อ, การรายงานผล, และการขอสลับวัน
